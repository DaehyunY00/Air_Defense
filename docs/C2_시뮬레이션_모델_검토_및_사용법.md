# K-JAMDS 시뮬레이터 — C2 관점 모델 검토 & 사용법

> ⚠️ **디스클레이머**: 본 문서가 다루는 모든 수치·좌표·확률은 공개자료(오픈소스) 기반의 **정책연구용 개념값**이며 실제 작전자료가 아닙니다. 모든 좌표는 도시 수준 개념좌표이고, KP-SAM(신궁)·천마(K-31)의 탄도탄 요격 불가, KAMDOC↔THAAD 미연동 등 제약이 모델에 반영되어 있습니다.

이 문서는 두 부분으로 구성됩니다.

- **[1부] C2 관점 모델 검토** — 통합방공(IADS) 지휘통제(C2) 이론의 관점에서 이 시뮬레이션이 무엇을·어떻게 모델링하는지, 그 강점과 한계를 정리
- **[2부] 사용법** — 웹 UI·딥링크·테스트를 실제로 구동하고 결과를 해석하는 방법

# 1부 — 통합방공 C2 관점의 모델 검토

## 1.1 이 시뮬레이터가 답하려는 C2 질문

한국형 합동방공체계(K-JAMDS)의 지휘통제 구조를 두 형상으로 놓고 비교한다.

| 형상 | C2 구조 | 핵심 특성 |
|---|---|---|
| **As-Is (분절형)** | 육군 ADC2A(군단 AOC·수방사 JAOC) ↔ 공군 MCRC·KAMDOC이 **데이터링크 미연동** | 항적 융합 부재, 육↔공 **음성 협조(≥180초)**, 이원화된 결심권, 중복교전·책임공백 |
| **To-Be (통합형)** | **JAMDC2(Track Fusion)** 융합 허브 경유 전 노드 데이터링크 연동 | Any Sensor–Best Shooter, AI 식별·자동 무기배정, 위협별 **사전승인 자동교전** |

핵심 정책 질문: *"위협 도착률(부하)이 C2 노드를 포화시키는 임계 구간(ρ≥0.9)에서, 통합형 C2가 분절형 대비 누수율·대기시간을 얼마나 개선하는가?"* — 이 개선폭이 K-JAMDS 투자 정당화의 근거다.

## 1.2 모델링 대상 (C2·센서·슈터 계층)

세 계층을 노드로 표현한다 (`js/data/nodes.js`).

- **C2 노드** (`category:'c2'`): KAOC(전구), MCRC·KAMDOC·JAMDC2(작전), 군단 AOC·수방사 JAOC(전술). 각 노드는 **M/M/c/K 대기행렬 서버**로 모델링 — 서버 수 `c`, 항적당 처리시간 `serviceTimeSec`(As-Is/To-Be 별도), 대기실 용량 `K`.
- **센서 노드** (`category:'sensor'`): 합동대공감시소, 방공관제레이더(동/서), 저고도레이더, 국지방공레이더(군단/수방사), E-737, 그린파인(탄도탄), 이지스 SPY-1D(동/서). 각각 `detects`(탐지 가능 위협 클래스)·`coverage`(담당 축선)를 가진다.
- **슈터 노드** (`category:'shooter'`): 전투기, 단거리방공(신궁·천마·비호·벌컨), 중거리(천궁), 미사일방어(천궁-II/PAC-3, L-SAM), SM-2. 교전채널 수·교전시간·`canEngage`(위협별 교전 가부, **제약조건 내장**)·`controlledBy`(통제 C2).

**C2 관점 강조점**: 무기의 교전 가능 여부(`canEngage`)와 통제 관계(`controlledBy`)가 데이터로 명시돼 있어, "누가 무엇을 쏠 수 있고 누구의 명령을 받는가"라는 지휘관계가 시뮬레이션 로직으로 강제된다. 예: 신궁/천마는 `srbm:false, mrl_large:false`로 탄도탄 교전이 원천 차단된다.

## 1.3 C2 프로세스 모델 — 9단계 파이프라인

DES 엔진(`js/engine/sim-engine.js`)은 개별 위협 객체를 다음 9단계에 흘려보낸다. 이는 교전순환(F2T2EA)·의사결정순환(OODA)·위협평가·무기할당(TEWA)과 대응된다.

```
1 탐지(Detect)        ─ 센서 스캔, 저탐지 위협은 항적소실→재획득 반복
2 추적생성(Track)     ─ 최속 보고링크로 담당 C2에 항적 전달
3 식별(Identify)      ┐
4 위협평가(TE)        ├ C2 서버 처리 — To-Be는 JAMDC2 융합 노드에서 집중 수행
5 무기-표적할당(WTA)  ┘
6 결심(Decision)      ─ 교전승인권자 (As-Is는 상위 제대 승인 필요)
7 교전협조/권한위임   ─ 육↔공 coord 홉 (As-Is 음성 지연·중복교전의 원천)
8 교전/요격명령       ─ 명령 링크 지연 후 무기 교전채널(M/M/c) 처리
9 BDA                 ─ 요격확률 판정 → 실패 시 재교전 피드백(폐루프, 상한 3회)
```

| 파이프라인 단계 | F2T2EA | OODA | TEWA |
|---|---|---|---|
| 1 탐지 / 2 추적 | Find, Fix, Track | Observe | Detection & Tracking |
| 3 식별 / 4 위협평가 | Target | Orient | Threat Evaluation |
| 5 WTA / 6 결심 / 7 협조 | Target → Engage | Decide | Weapon Assignment |
| 8 교전명령 / 9 BDA | Engage, Assess | Act | Engagement & Assessment |

**C2 관점의 핵심 설계**:

- **As-Is 협조 홉(7단계)**: 항적을 보유한 C2가 교전승인권자에 도달하려면 `coord` 링크 최단경로(BFS)를 타야 하는데, 육↔공 링크는 음성(180초)이다. 승인권자까지 경로가 아예 없으면 `responsibility_gap`(책임공백)으로 누수 처리된다.
- **To-Be 융합 경유**: C2에 도착한 항적은 `JAMDC2`로 다시 라우팅(`_onFusionArrive`)돼 융합·AI식별·WTA를 한 곳에서 처리하고, 대부분 위협이 `approvalLevel=null`(사전승인)이라 결심·협조 홉을 **생략**하고 바로 교전한다.
- **결심 지연의 치명성**: 탄도탄(`dwellSec` 90초)·순항미사일(120초)처럼 요격 가능 시간창이 짧은 위협은, As-Is 승인 지연이 곧 요격기회 상실로 직결된다 — 짧은 dwell + 긴 협조지연의 상호작용이 누수로 나타난다.

## 1.4 병목은 "도출"된다 — 하드코딩 금지 원칙

이 모델의 가장 중요한 방법론적 특징은 **병목 위치를 데이터에 고정하지 않는다**는 점이다. 병목은 세 입력의 상호작용에서 계산된다:

```
[시나리오 위협 도착률 λ]  ×  [모드별 C2 토폴로지]  ×  [노드 M/M/c/K 처리용량]
                              ↓
              이벤트 전개 → 관측 통계(ρ, 대기열, 드롭, 누수)
                              ↓
        임계값 초과 지점을 병목으로 "도출" (심각도순 정렬)
```

**병목 판정 임계값** (`ENV-RHO-THRESH-01`):

| 수준 | 조건 | C2 해석 |
|---|---|---|
| `normal` | ρ < 0.7 | 정상 처리 |
| `warn` | 0.7 ≤ ρ < 0.9 | 주의 — 수요 증가 시 대기 급증 위험 |
| `bottleneck` | ρ ≥ 0.9 | 병목 — 대기시간 비선형 폭증 구간 |
| `saturated` | 드롭 발생(K 초과) | 포화 — 항적/교전기회 상실(누수) |
| 통신병목 | 지연 ≥ 60초 **and** 전달중 체류 ≥ 1건 | 링크 병목 (Little's Law 기반) |
| 구조적 공백 | `no_sensor`/`no_shooter`/`responsibility_gap` | 탐지공백·교전수단부재·책임공백 |

같은 시나리오라도 강도 배수(`intensity`)나 모드(`asis`/`tobe`)를 바꾸면 병목 위치가 함께 이동한다. 통신병목 판정이 "지연 자체"가 아니라 "지연 × 유통량(체류 항적수)"인 점도 중요하다 — 저강도에서는 음성 링크조차 병목이 아니게 되어, 병목이 부하의 함수임을 드러낸다.

## 1.5 두 가지 분석 엔진

| 엔진 | 파일 | 방법 | 용도 |
|---|---|---|---|
| **해석적(Phase 1)** | `analysis/bottleneck.js` | 정상상태 M/M/c(Erlang-C) 근사, 부하를 그래프에 전파 | 빠른 개략 진단, 시나리오 탭 |
| **DES(Phase 2)** | `engine/sim-engine.js` | 이벤트 구동(이진 min-heap), 위협/노드 FSM, 관측통계 수집 | 실증적 병목·결과지표 |
| **Monte Carlo(Phase 3)** | `analysis/mc-runner.js` | DES 다중복제 + Welford CI 수렴·민감도 스윕 | 신뢰구간·통계적 유의성 |
| **임계 전환점(Phase 5)** | `analysis/transition.js` | 강도 스윕별 As-Is↔To-Be 격차 곡선 | ρ≥0.9 구간 개선폭 정량화 |

**재현성**: 모든 무작위성은 seed 기반 Mulberry32 RNG(`core/rng.js`)에서만 나오고, 동시 이벤트는 `(시각, 우선순위, 삽입순서)`로 결정론적으로 해소된다 → **동일 config는 항상 동일 결과**(딥링크로 공유·재현 가능).

## 1.6 C2 관점의 강점과 한계

**강점**

- 지휘관계(통제·승인·협조)를 링크 그래프로 명시하고, 협조경로를 BFS로 실제 탐색해 **책임공백**을 구조적으로 도출한다.
- As-Is 음성협조(180초)·To-Be 데이터링크(2초)·Link-16(12초) 등 **통신특성 차이**가 결과에 직접 반영된다.
- 위협별 요격 시간창(`dwellSec`)과 처리지연의 상호작용으로 "결심 속도가 왜 중요한가"를 정량화한다.
- 제약조건(신궁·천마 탄도탄 불가 등)이 로직으로 강제돼 비현실적 교전이 발생하지 않는다.
- 병목 하드코딩 금지 원칙으로 "병목은 부하의 함수"라는 주장을 반증 가능하게 만든다.

**한계 (해석 시 유의)**

- 모든 파라미터가 **개념값**이다. 절대 수치가 아니라 **As-Is↔To-Be 상대 비교**로 해석해야 한다.
- 물리(교전 기하·미사일 운동·요격 포락선)는 모델링하지 않는다 — `dwellSec`·`pk`로 추상화.
- 지리는 4개 축선(west/central/east/seoul) 개념 좌표로 단순화 — 실제 커버리지 중첩·사각은 근사.
- 융합 이득이 `dup_tobe = 1 + (dup-1)*0.3` 등 계수로 파라미터화 — 실제 융합 알고리즘 성능이 아님.
- 인적 요인(피로·판단오류·통신 혼선)은 서비스시간 분포에만 흡수돼 있고 개별 모델링은 없다.

자세한 검증 이력·타당성·한계는 `docs/vv-report.md`, 파라미터 근거는 `docs/params.md`를 참조한다.

# 2부 — 사용법

## 2.1 실행 (웹 UI)

정적 웹 페이지라 빌드가 필요 없다. CDN(Leaflet)의 타일 로딩 CORS를 피하려면 로컬 서버 권장:

```bash
cd "Air_Defense-main"
python3 -m http.server 8000 --bind 127.0.0.1
# 브라우저에서 http://localhost:8000 접속
# (--bind 없이 실행하면 터미널에 뜨는 IPv6 링크(http://[::]:8000/)가 클릭해도 안 열릴
#  수 있음 — 항상 위 http://localhost:8000 주소로 접속할 것)
```

외부 의존성은 Leaflet 1.9.4(CDN) 하나뿐이다. 폐쇄망 등 CDN 접근 불가 시 지도 탭은 자동으로 내장 SVG 개념도로 대체(graceful degradation)되고 나머지 기능은 동일하게 동작한다.

## 2.2 탭 구성 (개편: 실행 → 지도 시각화 → 결과창)

| 탭 | 무엇을 보는가 | 대응 엔진 |
|---|---|---|
| **시뮬레이션(sim)** | 지도 위 자산 배치·범위 링, [▶ 시작] → 위협궤적·노드 재고 링 애니메이션(백그라운드 DES·MC) → 종료 시 **결과창**(요약·비교·MC CI·병목·funnel·히트맵·노드표) | `sim-engine.js`(trace), `mc-runner.js` |
| **병목 분석(analysis)** | 노드 이용률·통신병목·타임라인·공백 도출 (정상상태 해석) | `bottleneck.js` |
| **Monte Carlo(mc)** | CI 수렴·통계적 유의성·민감도 토네이도·**임계 전환점** | `mc-runner.js`, `transition.js` |
| **근거자료·제약검증(data)** | 파라미터 출처·자산 범위(개념값), 제약 어서션 상시 검증 | `constraints.js` |

**체계 모드**는 상단의 단일 토글 스위치다: off = As-Is 분절형, on = To-Be 통합형.

## 2.3 핵심 파라미터

모든 탭이 공유하는 상태(해시가 단일 원천):

| 파라미터 | 의미 | 값 |
|---|---|---|
| `sc` | 시나리오 (KJADS 3대 문제 상황 1:1) | `sc1`(교전 중복·책임 공백 — 책임구역 경계 침투), `sc2`(무인기 8대 동시 남파 burst — 대응 실패), `sc3`(전략적 섞어쏘기 — 복합 동시 포화, 핵심) |
| `mode` | C2 형상 | `asis`(분절형) / `tobe`(통합형) |
| `x` | 강도 배수 | 전체 도착률 λ에 곱 (0.5~3.0) |
| `seed` | RNG 시드 | 정수 — 고정 시 재현 가능 |
| `dur` | 시뮬레이션 시간 | 초 (기본 1800 = 30분) |

## 2.4 딥링크 스킴 (상태 공유·재현)

```
#tab=<sim|analysis|mc|data>&sc=<ID>&mode=<asis|tobe>&x=<배수>&seed=<정수>&dur=<초>
```

구 딥링크의 `tab=map|scenario|des|playback`은 자동으로 `sim`으로 흡수된다.

예시:

- `index.html#tab=sim&sc=sc3&mode=asis&x=1.5&seed=12345` — 섞어쏘기 As-Is 1.5배 (시작 버튼으로 실행)
- `index.html#tab=analysis&sc=sc1&mode=asis` — 경계 침투 시나리오 해석 분석
- `index.html#tab=mc&sc=sc3&mode=asis&x=2` — Monte Carlo·임계 전환점

## 2.5 프로그래밍 방식 실행 (콘솔·Node)

브라우저 콘솔이나 Node에서 `window.KJ` 네임스페이스를 직접 호출할 수 있다.

```js
// 단일 DES 복제
const r = KJ.runDES({
  scenario: KJ.scenarioById('sc3'),
  mode: 'asis', intensity: 1.5, seed: 12345, endTimeSec: 1800
});
console.log(r.global.leakRate, r.bottlenecks);

// trace 모드 (재생/Gantt/시계열용 — 통계에는 영향 없음)
const rt = KJ.runDES({ scenario: KJ.scenarioById('sc3'), mode: 'tobe',
  intensity: 2, seed: 777, endTimeSec: 1800, trace: true, traceCap: 300 });
// rt.threatTraces (위협별 9단계 타임스탬프), rt.nodeSeries (노드 재고 시계열)

// Monte Carlo (95% CI 수렴 판정)
const mc = KJ.runMonteCarlo(
  { scenario: KJ.scenarioById('sc3'), mode: 'asis', intensity: 1.5, seed: 12345, endTimeSec: 1800 },
  { minReps: 30, maxReps: 500, tol: 0.01, primary: 'leakRate' });
```

## 2.7 권장 분석 워크플로우 (C2 분석가용)

1. **기준선 확인** — `sc1`을 강도 0.5×로 실행해 병목 0건(모든 노드 `normal`)을 확인. 병목이 부하의 함수임을 확인하는 출발점.
2. **문제 상황 재현** — `sc1`(경계 침투)에서 결과창의 **중복교전 위험 히트맵**(As-Is에서 위험 존재 → To-Be 0), `sc2`(무인기 burst)에서 낮은 격추율(저pk)·탐지 소실, `sc3`(섞어쏘기)에서 처리용량 포화를 각각 관찰.
3. **형상 비교** — 동일 `sc`·`x`·`seed`에서 모드 토글만 바꿔 재실행 — 결과창의 As-Is↔To-Be 비교표(동일 seed 단일 복제)와 백그라운드 MC CI로 대조.
4. **통계적 유의성** — 결과창(또는 MC 탭)에서 As-Is/To-Be의 95% CI가 겹치지 않는지 확인(개선이 표본변동이 아님을 검증).
5. **투자 정당화** — MC 탭의 **임계 전환점** 카드에서 As-Is C2 이용률이 0.9를 넘는 강도(`rho09CrossX`)와 그 이후 구간의 평균 개선폭(`postGapMean`)을 확인 — 이 구간이 통합 C2 가치의 핵심 논거.
6. **민감도 확인** — 토네이도에서 무엇이 누수율을 지배하는지 본다. `sc3`(포화)는 처리시간·강도가, `sc2`(무인기)는 요격확률이 지배적 — 시나리오별 제약 구조가 다름을 정량 확인.
7. **정성 관찰** — 시뮬레이션 탭의 지도 애니메이션에서 burst 동시 남파의 산개, 노드 재고 링이 붉게 두꺼워지는 시점(포화 진입)을 시각적으로 확인.

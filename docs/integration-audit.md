## G2 — 되돌리기 가능성 (Reversibility)
**핵심 발견: 전체 bit-exact 되돌리기는 CRN 때문에 원천 불가하다.** 기준선(`0468f10`)은 단일 RNG(`this.rng`가 도착·처리 공용)를 쓰지만, 현행은 **CRN**(`arrRng` 분리 — As-Is↔To-Be 짝지은 비교용)이다. CRN은 도착 스트림을 구조적으로 재배치하므로, 어떤 플래그 조합으로도 `0468f10`의 수치를 **완전 재현할 수 없다**. 이는 정직한 한계이며, 되돌리기는 세 층위로 제공한다:

1. **⑨ 런타임 플래그**(pkByShooter·leakCost·censorFix·timeoutSplit·pkCorrelated·salvo): 전부 OFF → **stage9 직전 지문과 36/36 완전 일치**(`reengage.test.js`). CRN 이후 기준선에 대한 bit-exact 되돌리기 — 증명 완료.
2. **`sensorPdFusion` 플래그(W6, 신규)**: OFF → 통합 이전 탐지식 `p=min(1, detectFactor×mult.detect)`(센서 Pd·융합 무시)로 복귀. 탐지 계층 **그리기 수 동일**(스캔당 raw 1회)이라 이 계층에선 bit-clean. 결정(reform)의 되돌리기를 증명.
3. **Git 층위 되돌리기**: W1~W4는 `0468f10`에서 각 브랜치가 **격리된 커밋 범위**다. `git checkout <브랜치直前>`으로 임의 역사 상태를 재현 가능(감사 추적은 git 로그). 런타임 플래그를 이 단계들에 소급하지 않은 이유: (a) CRN로 bit-exact가 어차피 불가, (b) 이미 병합·검증된 코드에 플래그를 심으면 회귀 위험이 이득을 초과, (c) git이 동일 감사 기능을 제공.

> **정직 보고**: 계획의 Gate 2 문구("모든 플래그 OFF = W0 이전 기준선 완전 동일")는 CRN 도입으로 **문자 그대로는 성립 불가**. 되돌리기의 *목적*(조작이 아님의 증명)은 위 3층으로 충족한다.

**개별 기여도(런타임 플래그로 격리된 것):**
- **⑨ Phase별 원장**(P0~P4, `vv-report §4-6`): 🔴 누수율 개선폭 +32.2%·구조실패 +33.7% — **지배 요인 단일 = P3 절단보정**(근거 B). 제외 시 전부 녹색. 편향 방향은 To-Be 유리이나 *정당한 교정*(As-Is가 파이프라인에 더 갇혀 미해결분 많음을 드러냄).
- **W6 sensorPdFusion**: ON↔OFF 탐지율 Δ **0.16pp**·교전지연 Δ **1.1s** — **거의 무영향**. 가장 두려워한 "하류 전체 이동" 웨이브가 실제론 집계 결론을 거의 안 움직인다(레버는 다센서 vs 단일센서 *대조*이지 집계치가 아님). `detect.test.js`의 축선별 대조가 실질 내용.
- **exchange 함정**: 기준선 To-Be 0.98>0.94(악화)는 "안 쏘면 exchange=0 최적" 함정의 발현(To-Be가 더 쏴서 나빠 보임). `defenseEfficiency`(⑨ Phase 2)가 이를 반전 — SC3 x2.5 As-Is 16.1% vs To-Be 65.6%. 결론은 exchange가 아니라 defenseEfficiency로 읽는다.
